# Obsidian 문서 생성 스킬

대화 내용, 분석 결과, 리서치 등을 옵시디언 마크다운 문서로 정리하여 볼트에 저장한다.

---

## 옵시디언 볼트 경로

```
/mnt/c/Soronprfbs/obsidian
```

---

## 저장 경로 결정

### 기본 경로

사용자가 경로를 지정하지 않으면 `1.Project/AI/` 하위에 저장한다.

1. `1.Project/AI/` 하위 기존 폴더 목록을 확인한다
2. 문서 주제와 일치하는 기존 폴더가 있으면 그곳에 저장한다
3. 적합한 폴더가 없으면 주제 기반으로 새 폴더를 생성한다
   - 폴더명: 영문 케밥케이스 (예: `harness-engineering/`, `api-design/`)

```bash
# 기존 폴더 확인
ls /mnt/c/Soronprfbs/obsidian/1.Project/AI/
```

### 사용자 지정 경로

사용자가 "3.Resource에 저장해줘", "Daily에 넣어줘" 등 경로를 지정하면 해당 경로에 저장한다.
볼트 루트(`/mnt/c/Soronprfbs/obsidian`) 기준 상대 경로로 해석한다.

## 파일명 규칙

- 한글 제목 사용 가능: `FastAPI DDD 아키텍처 설계.md`
- 공백 허용 (옵시디언 호환)
- 날짜 접두사 불필요 (frontmatter에 date 있음)
- 동일 이름 파일 존재 시 사용자에게 확인 후 덮어쓰기 또는 이름 변경

## 문서 구조

모든 문서는 아래 구조를 따른다:

```markdown
---
title: 문서 제목
date: YYYY-MM-DD
tags:
  - 태그1
  - 태그2
---

# 문서 제목

## 개요

[핵심 내용 1-3문장 요약]

## 본문

[주제에 맞게 섹션 구성]

## 참고

[관련 링크, 출처 등]
```

### frontmatter 규칙

- `title`: 문서 제목
- `date`: 문서 생성일 (YYYY-MM-DD)
- `tags`: 관련 태그 배열. 기존 볼트에서 사용 중인 태그와 일관성 유지

### 본문 작성 규칙

- 옵시디언 내부 링크(`[[문서명]]`) 활용 — 관련 문서가 존재할 가능성이 높으면 링크 생성
- 코드 블록에 언어 지정: ` ```python `, ` ```typescript ` 등
- 긴 문서는 목차 대신 명확한 헤딩 계층으로 구조화 (옵시디언이 자동 목차 제공)
- 표, 리스트, 콜아웃(`> [!note]`, `> [!warning]`) 적극 활용

## 다이어그램 / 아키텍처 시각화

문서 내용 중 다이어그램, 플로우차트, 아키텍처 구성도, 시퀀스 다이어그램 등
시각화가 필요한 부분이 있으면:

1. 문서 본문에서 해당 부분을 텍스트로 설명한다
2. 사용자에게 excalidraw 스킬로 시각화를 만들 것을 안내한다:
   > "이 아키텍처를 다이어그램으로 시각화하려면 excalidraw 스킬을 사용해주세요."
3. 직접 excalidraw 파일을 생성하지 않는다 — excalidraw 스킬의 역할이다

---

## 작업 순서

1. **주제 파악** — 문서화할 내용의 범위와 핵심을 파악한다
2. **경로 결정** — 기존 폴더 확인 후 저장 위치를 결정한다
3. **문서 작성** — frontmatter + 본문을 위 구조에 맞게 작성한다
4. **저장** — Write 도구로 파일을 생성한다
5. **결과 보고** — 저장 경로와 문서 개요를 사용자에게 알린다
